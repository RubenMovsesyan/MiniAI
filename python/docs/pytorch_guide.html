PyTorch: Match the C++ Model

Every primitive we hand-wrote in CUDA has a one-line PyTorch equivalent. This is the map — not the finished code. You assemble it in main.py.

The C++ engine is a few thousand lines. The same network in PyTorch is about thirty. PyTorch supplies the layers, the autograd, the loss, and the optimizer; you supply the architecture and the loop. Below, each row of the repo maps to the exact API to look up.

How to use this page. Each section names a PyTorch call and what it does, and points at the main.py TODO it fills. It deliberately does not paste a working model — assembling it is the exercise. If you get stuck, the official "Learn the Basics" tutorial covers every piece.

2. The model

Two linear layers with a ReLU between them. In PyTorch, a stack of layers is an nn.Module; the simplest way to build a straight-through stack is nn.Sequential. The forward pass — the affine map $\mathbf{z}=\mathbf{x}W+\mathbf{b}$ and the ReLU $\max(0,z)$ we derived by hand — is what nn.Linear and nn.ReLU already are.

Last layer emits logits, not probabilities. Exactly as in the C++ engine, the final layer is Identity — no softmax. Softmax lives inside the loss. If you add a nn.Softmax at the end and use CrossEntropyLoss, you apply softmax twice and the network trains badly. Leave it off.

Weight init. PyTorch's nn.Linear defaults to a Kaiming-uniform init that is already sane, so you can get away without touching it. To match the repo's He (kaiming_normal_, fan_in, ReLU nonlinearity) exactly, loop the nn.Linear modules and re-init their .weight; zero the .bias. As the init chapter showed, on a 2-layer net the scheme barely matters — but it's one line, so match it.

3. The loss — the one thing that bites everyone

This is the subtlety that took a whole chapter in the math docs. In the C++ engine the loss owns softmax and consumes raw logits; the fused gradient is $(\,\mathbf{p}-\mathbf{y})/N$ after the softmax Jacobian cancels the cross-entropy derivative. nn.CrossEntropyLoss is that exact object: it is LogSoftmax + NLLLoss fused, it takes logits, and it applies the same $1/N$ batch mean internally.

Two traps. (1) Do not softmax before the loss — pass logits. (2) CrossEntropyLoss wants integer class indices $\in\{0,\dots,9\}$, not one-hot vectors. That's why data.py keeps labels as int64 indices, unlike the C++ loader which one-hots. Pass y straight through.

You never write softmax, never write $\log$, never divide by $N$ — the three things we implemented by hand and gradient-checked. One object does all of it: loss = criterion(logits, y).

4. The training loop

The four-step dance is identical to train_step in the engine — only the names change. For each batch:

# the shape of it — fill in from the checklist
opt.zero_grad()          # = zero_grad(): clear accumulated grads
logits = model(xb)       # = forward()
loss   = criterion(logits, yb)
loss.backward()          # = backward(): autograd fills every .grad
opt.step()               # = update(): theta -= lr * grad
Autograd replaces the entire backward pass. The dense-layer gradients $dW=X^\top dZ$, $db=\mathrm{colsum}(dZ)$, $dX=dZ\,W^\top$, the ReLU gate — all of it, which we derived and coded in layer.cu, is what loss.backward() computes automatically by walking the graph PyTorch recorded during model(xb). You define forward; you get backward for free.

Device. Move the model and each batch to the GPU (.to("cuda")). The config's DEVICE is already read for you in main.py.

5. Evaluation

AccuracyMeter counted argmax(logits) == argmax(target) on-device. Same idea, one line — but wrap it in torch.no_grad() so PyTorch doesn't build a graph you won't backprop (saves memory and time). Since labels are indices here, you compare against y directly, not its argmax:

correct += (logits.argmax(1) == yb).sum().item()

6. The target to hit

With lr 0.1, batch 100, He init, 10–15 epochs, you should trace this curve — the one the C++ engine produced. Landing far below ~97% means a bug (double softmax, one-hot labels into CE, forgot zero_grad, or model still on CPU while data is on GPU).

Target — C++ engine, lr 0.1, batch 100, seed 42. Your PyTorch run should sit right on top of this.
Sanity checkpoints. Epoch 1 should already be ~92%. Loss should fall fast then flatten near 0.05. If epoch 1 is ~10% (chance) and loss sits at $\ln 10 \approx 2.30$, the model is outputting constant predictions — check the loss wiring first.
