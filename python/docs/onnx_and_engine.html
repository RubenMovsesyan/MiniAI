ONNX & the Inference Engine

A trained model is weights + a compute graph. ONNX is the standard file that carries both. This is how you produce one — and two ways to run it.

1. What ONNX actually is

Training produces two things: a graph of operations (matmul, add, relu…) and a set of learned tensors (the weights and biases). A framework like PyTorch keeps those in its own in-memory objects. To run the model somewhere else — a C++ service, a phone, a browser, your own CUDA engine — you need a portable, framework-neutral description of both. That's ONNX (Open Neural Network Exchange): a single .onnx file, encoded as protobuf, containing

  • a graph — a list of nodes, each naming an operator (Gemm, Relu, …), its inputs, and its outputs, wired by name;
  • initializers — the constant tensors (your trained $W$ and $b$) baked in;
  • value info — the input/output names, shapes, and dtypes;
  • an opset version — which revision of the operator definitions this graph targets, so a runtime knows exactly how each op behaves.
The mental model. ONNX is a frozen dataflow graph. No Python, no training loop, no autograd — just "here are the ops, here are the weights, here's how they connect." Anything that can read the graph and implement the ops can run the model.

2. Our model as an ONNX graph

The 784→128→10 classifier is tiny. Each nn.Linear is one Gemm node (general matrix multiply with bias: $\alpha XW + \beta b$ — exactly our affine map), and the middle nonlinearity is one Relu node:

input
(N, 784) → Gemm
W1 784×128, b1 → Relu → Gemm
W2 128×10, b2 → logits
(N, 10)
No softmax in the graph — on purpose. Just like the training engine, where the loss owned softmax and the last layer emitted logits, the inference graph can stop at logits. The prediction is $\arg\max$ over the 10 outputs, and softmax is monotonic — it doesn't change which entry is largest. So you skip it and read the class off the logits. Add a trailing Softmax node only if a downstream consumer needs calibrated probabilities.

3. Producing the .onnx — two routes

Route A — hand-build the graph (from any trained weights)

Because ONNX is just a data structure, you can construct it directly with the onnx Python API: make a Gemm node, a Relu node, another Gemm, attach your weight arrays as initializers, and save. This is the route you'd take to export the C++ engine's weights without going through PyTorch at all — dump $W_1,b_1,W_2,b_2$ from the CUDA model to .npy, then:

from onnx import helper, TensorProto, numpy_helper
n1 = helper.make_node("Gemm", ["input","W1","b1"], ["h"])
n2 = helper.make_node("Relu", ["h"], ["a"])
n3 = helper.make_node("Gemm", ["a","W2","b2"], ["logits"])
graph = helper.make_graph([n1,n2,n3], "mnist",
    inputs=[helper.make_tensor_value_info("input", TensorProto.FLOAT, ["N",784])],
    outputs=[helper.make_tensor_value_info("logits", TensorProto.FLOAT, ["N",10])],
    initializer=[numpy_helper.from_array(W1,"W1"), # ...b1, W2, b2])
onnx.save(helper.make_model(graph), "mnist.onnx")

Note Gemm's transB attribute — PyTorch stores a linear layer's weight as (out, in), so exports set transB=1. Our C++ layout is (in, out), so a hand-build wouldn't transpose. Small detail, easy to get backwards — check it against a known input.

Route B — torch.onnx.export (the one you'll use)

Once the model is in PyTorch, export is a single call. PyTorch traces a forward pass with a dummy input, records the ops, and writes the graph + weights for you. This is the export_onnx TODO in main.py:

import torch
dummy = torch.randn(1, 784, device="cuda")
torch.onnx.export(model, dummy, "mnist.onnx",
    input_names=["input"], output_names=["logits"],
    dynamic_axes={"input":{0:"N"}, "logits":{0:"N"}})  # variable batch

dynamic_axes marks the batch dimension as variable so the same file serves one image or ten thousand. Validate the result with onnx.checker.check_model("mnist.onnx").

4. Running it — the engine

Route A — onnxruntime (build this now)

ONNX Runtime is a production inference engine that loads an .onnx file and executes its graph, with a CUDA backend (CUDAExecutionProvider) that runs it on the same GPU. It's already in requirements.txt. Loading and running is a few lines:

import onnxruntime as ort, numpy as np
sess = ort.InferenceSession("mnist.onnx",
    providers=["CUDAExecutionProvider", "CPUExecutionProvider"])
logits = sess.run(None, {"input": x.astype(np.float32)})[0]
pred = logits.argmax(1)
The correctness check that matters. Run the same test batch through the PyTorch model and through onnxruntime and assert the logits match (np.allclose(torch_out, ort_out, atol=1e-4)). If they agree, the export is faithful and the ONNX model will score the same ~97.6% — the export changed the container, not the math. For a tiny MLP, CPU inference is instant, so a CPU-only onnxruntime is a fine fallback if the GPU wheel doesn't resolve.

Route B — your own C++/CUDA engine (a future from-scratch project)

In the spirit of the whole repo, you could skip onnxruntime and run the graph on your own kernels. The .onnx file is protobuf, so an engine would:

  1. Parse the file with the ONNX protobuf schema (onnx.proto) → a list of nodes and initializer tensors.
  2. Upload initializers — turn each weight tensor into a Matrix on the GPU (one cudaMemcpy, the src/io pattern).
  3. Topologically order the nodes (this graph is already a straight line) and walk them, keeping a name → Matrix table of intermediate results.
  4. Dispatch each op to a kernel you already have: Gemm → matrix.matmul + rowAdd, Relu → nn::relu. That's the entire operator set this network needs — two ops.
mnist.onnx→ protobuf parse→ op dispatch table
Gemm→matmul · Relu→relu→ src/matrix + src/nn kernels→ logits
Scope. A general ONNX engine (arbitrary ops, opset coverage, dynamic shapes) is a large project — that's why onnxruntime exists. But a minimal engine that runs exactly this graph is small: a protobuf parse plus a two-entry dispatch table over kernels you've already written and tested. It's a natural next from-scratch milestone once the PyTorch model and its ONNX export exist — a real target to run against.
